üì± Mobile App Issues

‚ö†Ô∏è App Won't Start / Metro Bundle Error

CRITICAL

Symptoms: Red error screen, "Bundle failed to load" message

‚úÖ Solution
  1. Clear metro cache: npm start -- --clear
  2. Delete node_modules & reinstall: rm -rf node_modules && npm install
  3. Restart metro bundler: npm start
  4. In Expo, press 'r' to reload

‚ö†Ô∏è Chat Not Responding / AI Concierge Hanging

CRITICAL

Symptoms: Chat screen loads but no response, spinning indicator

‚úÖ Solution Steps
  1. Check Backend API: curl http://localhost:8000/health Should return healthy status
  2. Verify Together AI Key: echo $TOGETHER_API_KEY Should show a valid key (not empty)
  3. Check Backend Logs: docker logs swipesavvy-ai-agents Look for 401 or connection errors
  4. Restart Backend: docker restart swipesavvy-ai-agents
  5. Test Chat Endpoint Directly: curl -X POST http://localhost:8000/concierge/api/v1/chat \ -H "Authorization: Bearer YOUR_TOKEN" \ -H "Content-Type: application/json" \ -d '{"message": "Hello"}'

‚ö†Ô∏è Authentication Failed / Login Issues

HIGH

Symptoms: Cannot login, "Invalid credentials" error, infinite loading

‚úÖ Solution Steps
  1. Clear app cache: Long-press app ‚Üí App info ‚Üí Storage ‚Üí Clear Cache
  2. Verify backend auth service is running
  3. Check environment variables for JWT secret
  4. Reset test user in database: psql -U postgres -d swipesavvy \ -c "UPDATE users SET password_reset=true WHERE email='test@example.com';"
  5. Restart app and try login

‚ö†Ô∏è Balance Not Updating / Wallet Shows Wrong Amount

HIGH

Symptoms: Balance stuck, doesn't update after transaction

‚úÖ Solution Steps
  1. Pull down to refresh (force manual sync)
  2. Restart app completely
  3. Check database directly: psql -U postgres -d swipesavvy \ -c "SELECT id, balance FROM accounts WHERE user_id='USER_ID';"
  4. If database balance is correct, clear Redis cache: redis-cli FLUSHDB
  5. Restart backend to reload cache

‚ö†Ô∏è Notifications Not Arriving

HIGH

Symptoms: Campaign notifications, reward alerts not showing

‚úÖ Solution Steps
  1. Verify Firebase setup: cat app.json | grep expo.plugins Should include Firebase plugin
  2. Check notification permissions on device
  3. Verify FCM token is registered: psql -U postgres -d swipesavvy \ -c "SELECT fcm_token FROM users WHERE id='USER_ID';"
  4. Test sending notification manually
  5. Check notification logs in Firebase console

‚öôÔ∏è Backend API Issues

‚ö†Ô∏è API Returns 500 Error / Internal Server Error

CRITICAL

Symptoms: API endpoints return 500, logs show exceptions

‚úÖ Solution Steps
  1. Check backend logs: docker logs swipesavvy-ai-agents | tail -50
  2. Look for database connection errors or missing tables
  3. Verify database schema is up to date: cd swipesavvy-ai-agents && alembic upgrade head
  4. Restart backend: docker restart swipesavvy-ai-agents
  5. If still failing, check environment variables are set

‚ö†Ô∏è Database Connection Refused / Cannot Connect

CRITICAL

Symptoms: "Cannot connect to database", connection timeout

‚úÖ Solution Steps
  1. Check if PostgreSQL is running: docker ps | grep postgres
  2. If not running, start it: docker-compose up -d postgres
  3. Verify database exists: psql -U postgres -l | grep swipesavvy
  4. Check connection credentials in .env
  5. Try direct connection: psql -h localhost -U postgres -d swipesavvy
  6. If password issues, reset: psql -U postgres -c "ALTER USER postgres PASSWORD 'newpassword';"

‚ö†Ô∏è Campaign API Endpoints Return 401 Unauthorized

HIGH

Symptoms: Campaign endpoints reject requests even with token

‚úÖ Solution Steps
  1. Verify JWT token is valid: curl -X POST http://localhost:8000/auth/token \ -H "Content-Type: application/json" \ -d '{"email":"admin@example.com","password":"password"}'
  2. Check token expiry (default 24 hours)
  3. Verify admin user exists and has admin role: psql -U postgres -d swipesavvy \ -c "SELECT id, email, role FROM users WHERE email='admin@example.com';"
  4. If missing admin role: psql -U postgres -d swipesavvy \ -c "UPDATE users SET role='admin' WHERE email='admin@example.com';"
  5. Restart backend to clear cache

‚ö†Ô∏è Analytics Data Not Showing / Empty Dashboard

MEDIUM

Symptoms: Analytics dashboard blank, no campaign metrics

‚úÖ Solution Steps
  1. Verify analytics tables exist: psql -U postgres -d swipesavvy -c \ "\dt analytics.campaign_analytics_daily"
  2. Check if data is being populated: psql -U postgres -d swipesavvy \ -c "SELECT COUNT(*) FROM analytics.campaign_analytics_daily;"
  3. If count is 0, manually trigger analytics job: curl -X POST http://localhost:8000/api/analytics/refresh
  4. Verify cron jobs are configured (pg_cron extension)
  5. Check backend logs for analytics errors

üóÑÔ∏è Database Issues

‚ö†Ô∏è Schema Missing Tables / Alembic Migration Failed

CRITICAL

Symptoms: "table does not exist" errors, migration incomplete

‚úÖ Solution Steps
  1. Check current migration version: cd swipesavvy-ai-agents && alembic current
  2. View migration history: alembic history
  3. Run pending migrations: alembic upgrade head
  4. If migration fails, check logs: alembic upgrade head --verbose
  5. If still issues, run schema setup directly: psql -U postgres -d swipesavvy \ -f AI_MARKETING_ANALYTICS_MVP_01_DATABASE_SETUP.sql

‚ö†Ô∏è Rewards Ledger Not Recording Points / Incorrect Balances

HIGH

Symptoms: Points not added to ledger, balance wrong

‚úÖ Solution Steps
  1. Check if rewards table exists: psql -U postgres -d swipesavvy \ -c "SELECT * FROM rewards_ledger LIMIT 5;"
  2. Verify user's current points: psql -U postgres -d swipesavvy \ -c "SELECT SUM(points) FROM rewards_ledger WHERE user_id='USER_ID';"
  3. Check if transactions are triggering rewards: psql -U postgres -d swipesavvy \ -c "SELECT * FROM transactions WHERE user_id='USER_ID' ORDER BY created_at DESC LIMIT 5;"
  4. Verify rewards service is running
  5. Manually trigger reward calculation

‚ö†Ô∏è Disk Space Running Out / Database Growing Too Large

HIGH

Symptoms: Database performance slow, writes fail, out of space

‚úÖ Solution Steps
  1. Check database size: psql -U postgres -c "SELECT pg_database.datname, pg_size_pretty(pg_database_size(pg_database.datname)) FROM pg_database WHERE datname='swipesavvy';"
  2. Check largest tables: psql -U postgres -d swipesavvy -c "SELECT schemaname, tablename, pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) FROM pg_tables ORDER BY pg_total_relation_size(schemaname||'.'||tablename) DESC LIMIT 10;"
  3. Archive old data if needed
  4. Run VACUUM to reclaim space: psql -U postgres -d swipesavvy -c "VACUUM ANALYZE;"
  5. Increase disk allocation

‚ö° Performance Issues

‚ö†Ô∏è API Response Slow / High Latency

HIGH

Symptoms: API calls take > 500ms, endpoints timing out

‚úÖ Solution Steps
  1. Check backend resource usage: docker stats swipesavvy-ai-agents
  2. Check database query performance: psql -U postgres -d swipesavvy \ -c "SELECT query, calls, total_time FROM pg_stat_statements ORDER BY total_time DESC LIMIT 10;"
  3. Check for missing indexes: psql -U postgres -d swipesavvy \ -c "SELECT schemaname, tablename FROM pg_tables WHERE schemaname='public' AND indexname IS NULL;"
  4. Clear Redis cache if stale: redis-cli FLUSHALL
  5. Increase backend replicas if horizontally scalable

‚ö†Ô∏è High Memory Usage / OutOfMemory Errors

HIGH

Symptoms: Application crashes, memory full, OOM kills

‚úÖ Solution Steps
  1. Check current memory usage: docker stats
  2. Restart container to clear memory: docker restart swipesavvy-ai-agents
  3. Increase allocated memory in docker-compose.yml: services: backend: mem_limit: 2g memswap_limit: 2g
  4. Look for memory leaks in logs
  5. Optimize database queries

‚ö†Ô∏è Campaign Notifications Sending Slowly / Batch Processing Delays

MEDIUM

Symptoms: Campaigns take hours to send, queue backed up

‚úÖ Solution Steps
  1. Check Celery queue: celery -A swipesavvy inspect active
  2. Monitor queue length: redis-cli LLEN celery
  3. Increase number of workers: docker-compose scale celery=4
  4. Check notification service logs
  5. Optimize batch processing size

üö® Escalation Procedures

When Issues Persist After Troubleshooting

  • Collect diagnostic information:
    • Backend logs (last 100 lines)
    • Database error logs
    • System metrics (CPU, memory, disk)
    • Network connectivity test results
    • Exact error messages and stack traces
  • Attempt rollback to previous working version
  • Contact development team with collected diagnostics
  • Document issue in ticketing system
  • For critical production issues, activate incident response
